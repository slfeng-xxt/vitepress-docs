# HarnessEngineering

## 1.什么是Harness

Harness Engineering是指围绕 AI Agent（特别是 Coding Agent）设计和构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。它解决的核心问题是：当 AI Agent 拥有了强大的代码生成能力后，如何确保其输出的可靠性、一致性和长期可维护性。

"Harness"本意是马具——缰绳、鞍具那一套东西，把马的力气引到正确方向上。拿来类比 AI Agent 挺合适：LLM 就像一匹蛮力十足但方向感不太行的马，跑得快但容易跑偏.

## 2.三层工程

Harness Engineering 并不是凭空出现的，它是 Prompt Engineering 和 Context Engineering 的自然延伸。三者构成嵌套关系：

```text
Phil Schmid 打了个比方：模型是 CPU，Harness 是操作系统——CPU 再强，OS 拉胯也白搭。
mtrajan 的区分更直接：Context Engineering 管的是"给 Agent 看什么"，Harness Engineering 管的是"系统怎么防崩、怎么量化、怎么修".
```

## 3. 核心哲学：为什么需要 Harness？

在 AI Agent 能够自主操作数据库、修改代码的今天，光靠 Prompt 这种“软约束”根本防不住它把生产环境搞崩。Harness Engineering 旨在解决以下核心矛盾：

- 无限状态 vs 有限 Token：物理世界的状态是无限的，但模型的 Context Window 是有限且昂贵的。Harness 必须建立高效的双向映射机制。
- 非确定性 vs 确定性工程：模型是概率性的，但工程要求确定性。Harness 必须在架构层面让模型“做不到”越界行为。
- 熵增 vs 熵管理：AI 生成的代码会以不同于人类的方式积累“技术债”。必须持续对抗信息债和代码熵。

## 4. 四大核心支柱

### 4.1. 上下文架构

智能体只能推理它能“看到”的内容。存储在 Slack 或工程师头脑中的知识，除非刻意导入，否则不可见。
渐进式披露：给智能体一张地图，而不是 1000 页手册。

多层级配置体系：

- 全局层：编码风格、快捷习惯。
- 项目层：架构规范、技术栈说明。
- 子目录层：/docs 格式要求、/tests 测试规范。
- AGENTS.md：Agent 的活文档。不是一次性写完就忘，而是反馈循环：每当 Agent 犯错，更新此文档。

### 4.2. 智能体专业化

- 限定作用域：不要给模糊的高层目标，分解为明确定义的小任务。
- 受限工具集：根据当前任务提供最小必要工具集，单智能体不需要访问所有工具。
- 深度优先：设计 → 代码 → 评审 → 测试。

### 4.3. 持久记忆

- 基于文件系统：不依赖对话历史的临时记忆，将关键状态存储在文件中（如 claude-progress.txt）。
- 跨会话连续性：通过 Git 历史记录和结构化文件实现多轮任务状态传递。
- 状态分离原则：把 LLM 视为无状态的 CPU，复杂的业务会话状态必须由 Harness 控制的外部引擎（Redis/DB）维护。

### 4.4. 结构化执行

将思考与执行分离。所有团队都施加了刻意的执行序列：

- 理解 → 规划 → 执行 → 验证
- 研究：探索代码库，只读访问。
- 规划：分解任务，制定计划（人工检查点：审查计划远比审查代码快）。
- 执行：基于验证过的计划，限定范围写入。
- 验证：自动化测试 + Linter + 人类审查。

## 5. 关键工程机制

### 5.1. 机械不变量

不要指望 AI “理解”架构规则，要用代码强制执行。

- Linter 即指令：自定义 Linter 在报错时直接包含修复方法，Agent 遇到违规时同时获得“教学”。
- 依赖方向强制：例如 Types → Config → Repo → Service，任何违反方向的代码被自动阻止。

### 5.2. 熵管理与“垃圾回收”

AI 生成的代码会积累“熵”。

- 定期清理：运行“垃圾回收 Agent”，扫描文档不一致、架构违规、冗余代码。
- 对话归档：压缩上下文前先落盘，确保完整历史可审计、可回放，不丢信息只移出活跃上下文。

### 5.3. 任务调度与硬约束

- TodoWrite + Nag 机制：同一时间只有 1 个 in_progress；超过 3 轮未更新，注入 reminder 作为硬提醒（在 tool_result 中，模型无法忽略）。
- 操作审批门控：高风险操作（删除文件、修改核心模块）必须审批。Reject 必须携带 Feedback，形成“拒绝→改进→重提交”闭环。
- Shutdown 握手：不直接杀线程，发送请求等待 Agent 在合适时机优雅退出，防止文件半写。

## 6.Function Calling 的生命周期

这是 Harness 最脆弱的环节，必须设计降级路径：

| 阶段 | 动作 | 失败处理策略 |
| :--- | :--- | :--- |
| Schema 序列化 | 将工具定义注入 Prompt | 确保 JSON Schema 准确，避免模型“幻觉”参数 |
| 触发生成 | 模型输出工具调用文本 | 监控 Token 消耗，防止死循环 |
| 反序列化 | Harness 解析 JSON | 重试：报错 `Invalid JSON` 要求重生成；回退：转自然语言指令 |
| 执行 | 物理执行器运行 | 超时/报错：注入错误日志，引导 Agent 反思或请求人工 |
| 观测注入 | 结果塞回 Prompt | 强制分隔符 `[Tool_Output]`，避免“大海捞针” |

## 8. 常见陷阱与术语

| 术语 | 定义/现象 | 解决方案 |
| :--- | :--- | :--- |
| Context Rot | 上下文窗口填满，性能下降 | 渐进式披露，对话归档，摘要压缩 |
| Context Pollution | 上下文中存在过多无关/冲突信息 | 严格的注入边界，Token 预算分配 |
| Doom Loops | 智能体在同一个错误方法上反复尝试 | 最大重试次数，Nag 机制，人工介入门控 |
| Entropy | AI 生成的代码积累的技术债 | 垃圾回收 Agent，机械不变量 (Linter) |
| Lost in the Middle | 关键信息在长上下文中间被忽略 | 关键指令放首尾，强分隔符，算分排序 |

## 9. 总结

Harness Engineering 的本质是承认模型的局限性，并用工程手段弥补它。

- 不要试图用 Prompt 解决架构问题，用 Linter 和 CI。
- 不要指望模型记住状态，用 文件系统 和 数据库。
- 不要相信模型会自觉，用 结构化执行 和 审批门控。

```text
一句话心法：把 LLM 当作一个无状态、易出错、但极具创造力的 CPU，Harness 就是它的操作系统、内存管理和安全沙箱。
```
