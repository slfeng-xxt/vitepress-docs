## 2. Token

:::tip
openAI提供了文本切分token的工具[tokenizer](https://platform.openai.com/tokenizer)

Token: 大模型处理文本的 **「最基本单位」**，并不是词

1个token = 0.75个`英文单词` = 1.5-2个`汉字`;
40万token 大约等于 60-80万汉字或者30万英文单词

- todo：token生成机制是什么？

:::

- 编码：将用户输入文字切分成`token`，并映射为数字，最后将数字链表输入给大模型
- 解码：大模型将数字链表转化为文字

## 4. [RAG](/AI/RAG.md)

ß- 检索增强生成（Retrieval Augmented Generation）

:::tip
将大模型应用于实际业务场景时:

- 知识的局限性：大模型自身的知识完全源于训练数据，而现有的主流大模型（deepseek、文心一言、通义千问…）的训练集基本都是构建于网络公开的数据，对于一些实时性的、非公开的或私域的数据是没有。

- 幻觉问题：所有的深度学习模型的底层原理都是**基于数学概率**，模型输出实质上是一系列数值运算，大模型也不例外，所以它经常会**一本正经地胡说八道**，尤其是在大模型自身不具备某一方面的知识或不擅长的任务场景。

- 数据安全性：对于企业来说，数据安全至关重要，没有企业愿意承担数据泄露的风险，尤其是大公司，没有人将私域数据上传第三方平台进行训练会推理。这也导致完全依赖通用大模型自身能力的应用方案不得不在数据安全和效果方面进行取舍。

而RAG就是解决上述问题的有效方案

:::

### 4.1 功能

RAG就是通过检索获取相关的知识并将其融入Prompt，让大模型能够参考相应的知识从而给出合理回答

## 5. Prompt

:::tip
Prompt：大模型接收的具体问题或指令，用于引导大模型生成内容
:::

### 5.1 Prompt Engineering（略）

- 作用：提高大模型回答问题的准确性和相关性

- 重要性：Prompt是用户与AI之间的桥梁，设计良好的Prompt能够引导大模型生成高质量的内容

### 5.2 User Prompt（用户输入）

- 用户输入的指令或问题(具体要实现的任务描述)

### 5.3 System Prompt（后台配置）

- 系统输入的指令或问题（人设和做事规则制定），会影响大模型的输出行为

## 6. Tool（函数）

- 作用：让大模型调用外部工具，获取外部数据或执行特定任务

- 重要性：Tool能够增强大模型的实用性和灵活性，使其能够处理更复杂的任务

- 例如：天气查询工具的使用

```text
实现流程：

用户输入 --> 平台(需要用到的tools) --> 大模型（返回天气查询工具以及相关参数描述的json） --> 调用Tool（天气查询工具） --> 返回天气信息（字符串描述） 
--> 大模型（拿到信息后转化为人话输出） --> 平台（窗口输出内容） --> 用户（看到天气结果）

总结：

- 大模型：选择要使用的工具，并生成工具的参数描述；归纳总结

- 工具：查询天气的API

- 平台：串联这个流程
```

## 7. [MCP](/AI/MCP.md)

- （Model Context Protool）
- 工具接入平台的过程中需要一套规范---**统一的工具接入格式的标准协议**。（类似手机充电口的统一，type-c接口。。。）
- mcp可以接入外部工具，并生成工具的参数描述，让大模型能够调用外部工具

- 举例说明：

```text
用户输入：查询附近天气如何，如果下雨的话，我想附近超市买把伞

工具： 定位  天气  商铺
```

## 11. 新的编程范式

Prompt Engineering ======》  Loop Engineering

- PE 是命令式——人类主动给 AI 写指令，AI 执行后返回结果，人再介入。
- LE 是声明式——人类设计一套系统，由系统自动去 prompt AI、获取反馈、再触发下一轮，人只需定义目标和验收标准。
- 两者的技能要求不同，不是进阶关系，而是方向的转变。

Q：LE 适合什么类型的任务？
A：适合三类：
有明确目标且可自动验证结果的任务（如跑测试并修复失败）、高重复性但需要智能判断的任务（如扫描 issue 并分类）、人工处理成本远超 token 成本的任务。
反之，一次性的、创意性强的、结果难以量化的任务，用普通 prompt 更合适

Q：普通开发者现在就能用 LE 吗？
A：可以，但建议先打好基础。Claude Code 的 /loop 和 /goal 命令已具备 LE 的基本要素。但在上 Loop 之前，先确保你能写出可靠的单次 prompt——LE 会把 prompt 的问题放大，而不是掩盖它。

Q：LE 最大的风险是什么？
A：跑偏后难发现、难修复。一旦循环运行了几十轮才发现方向错了，排查成本极高。建议早期设置明确的停止条件、保留每轮的中间结果、并在关键节点插入人工确认节点。

总结：

- Loop Engineering 最深层的影响，不在于工具，而在于开发者的工作内容正在发生结构性转变。
- 过去是：理解需求 → 分解任务 → 写代码 → 调试。
- LE 之后变成：理解需求 → 设计循环规则 → 定义验收标准 → 观察并调优系统。
- 这更像一个系统架构师的工作，而不是一个执行层的工程师。能把需求翻译成精准的循环目标、能设计合理的质检标准、能判断系统什么时候跑偏——这些能力，比"会写好 prompt"要难得多，也有价值得多。
- 所以，LE 不是让开发者可以偷懒的工具，而是把开发者的工作提升到了更抽象的层次。入场门槛更高了，不是更低了。
