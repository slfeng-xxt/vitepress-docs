# 优化

:::tip
1.打开速度变快 -----> 首屏加载优化

2.再次打开速度变快 -----> 缓存优化

3.操作流畅 -----> 渲染优化

4.动画流畅 -----> 长任务拆分
:::

## 1.首屏加载优化

:::warning
存在情况：页面打开时，首屏加载时间过长，导致用户体验差
:::

有什么指标可以来衡量？**相关指标**如下

- 0.FP（First Paint）首次绘制，页面从开始加载到页面开始绘制的任何部分的时间，
- 1.FCP（First Contentful Paint）首次**内容绘制**，页面从开始加载到页面内容的任何部分在屏幕上渲染的时间(fp到fcp的时间差，就是白屏时间)
- 2.FMP（First Meaningful Paint）首次**有意义绘制**，页面主要内容出现在屏幕上的时间，比如首屏内容全部渲染出来（MutationObserver来做性能采集）
- 3.LCP（Largest Contentful Paint）**最大内容绘制**，最大内容元素在屏幕上完成渲染的时间

**其他指标**如下

- 4.INP（Interaction to Next Paint）从用户首次与页面交互到页面下一次**重绘**的时间
- 5.TTI（Time to Interactive）页面首次可交互时间(ssr中优化指标)
- 6.TBT（Total Blocking Time）**总阻塞时间**，FCP到TTI的时间差，TTI（Time to Interactive）页面首次可交互时间
- 7.CLS（Cumulative Layout Shift）累计布局偏移，页面内容在加载期间发生的移动次数(重排/重绘)
- 8.TTFB（Time to First Byte）首字节时间，从发起请求到接收到响应数据第一个字节的时间

### 1.1 优化方向-体积

- 1.**图片**优化，采用webp,图片尺寸（在合适的容器内使用合适的图片，1倍图，2倍图，3倍图）
- 2.**字体**文件瘦身，字体子集化（用到哪些字，最后只生成对应的字体文件）[Fontmin](https://ecomfe.github.io/fontmin/) - (功能：1.提取部分字体
2.转换字体格式3.生成 webfont 和对应 CSS 样式)
- 3.懒加载资源，**图片懒加载**，**脚本懒加载**，**组件懒加载**
- 4.css、javascript文件压缩，**tree shaking**，**代码分割**，**cdn**，**http2**，**gzip**，**缓存**，**预加载**，**预渲染**，**ssr**，**pwa**

## 2.动画卡顿

:::tip
为什么会卡顿？单线程，js执行和页面渲染是互斥的，js执行时，页面渲染是停止的，反之亦然。如果js执行时间过长，页面渲染就会卡顿

ps: 屏幕刷新率：60HZ，1秒60帧，每帧16.7ms，如果js执行时间超过16.7ms，就会导致卡顿(具体查看performance中的Frames中标红的，该区域就是js执行占用了大量的时间，超过16.7ms后没有渲染，就会导致卡顿)
:::

### 2.1 优化方向

#### 2.1.1 减少主线程阻塞

- 优化js执行，部分长任务拆分，使用requestIdleCallback，requestAnimationFrame，webworker（复杂的计算）
- **避免大量计算**，**避免大量重排重绘**，**避免大量dom操作**，**避免大量定时器**，**避免大量事件监听**，**避免大量ajax请求**

#### 2.1.2 GPU加速

- 使用css3动画，开启硬件加速，如transform，opacity，filter等
- 避免引起重排的属性，定位 left(交互多的用transform替换实现)
- 选择合适的帧率，requestAnimationFrame的回调函数中，尽量只做当前帧的计算，不要做下一帧的计算，下一帧的计算放到下一帧的回调函数中
- 使用requestAnimationFrame代替setTimeout，setInterval，requestAnimationFrame的回调函数会在每一帧开始时执行，而setTimeout，setInterval的回调函数会在指定时间后执行，如果指定时间过长，就会导致卡顿
- 节流防抖，减少回调函数的执行次数，减少回调函数的执行时间
